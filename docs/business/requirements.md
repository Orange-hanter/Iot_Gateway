## Видение продукта

**Gateway** — универсальный модуль сбора данных, разворачиваемый на edge-уровне предприятия (промышленные мини-ПК). Система решает проблему фрагментированной интеграции разнородного оборудования в цехах, предоставляя единый, стабильный и масштабируемый интерфейс для нашей платформы визуализации данных.

### Модель поставки

- Поставляется **в сборе с железом** как готовое решение
- Развёртывается **on-premise** — только в периметре предприятия
- **Мультитенантность не требуется** — один экземпляр = одно предприятие
- Подключение к внешним системам (MES/ERP) рассматривается как **будущее расширение**, не обязательный функционал

### Ключевая ценность

Предприятие получает **«единое окно»** для всех данных с оборудования:
- Данные с любого оборудования отображаются на наших дашбордах без кастомной разработки
- Подключение нового устройства — конфигурация в Node-RED, не написание кода
- Бизнес-логика и тревоги настраиваются инженерами визуально
- Данные не теряются при сбоях сети (store-and-forward)

## Функциональные требования

### 1. Сбор данных

- **FR-1.1** Поддержка до 100 IIoT-устройств одновременно *(текущий потолок, с планированием роста)*
- **FR-1.2** Протоколы по приоритету:
  1. **Modbus TCP/RTU** — первый клиент, целевой протокол
  2. **MQTT** — реализовано в пилоте
  3. **HTTP/REST** — реализовано в пилоте
  4. OPC UA, Sparkplug B — будущее расширение
- **FR-1.3** Автоматическая регистрация новых устройств
- **FR-1.4** Парсинг пакетов с множеством сенсоров (float, int, bool, string)
- **FR-1.5** Поддержка QoS для MQTT (уровни 0/1/2)
- **FR-1.6** Подключение нового типа устройства через Node-RED без перезапуска ядра
- **FR-1.7** Целевая нагрузка первого клиента: ~30 сенсоров

### 2. Буферизация и надёжность

- **FR-2.1** Хранение данных локально при потере связи (store-and-forward)
- **FR-2.2** Настраиваемый объём буфера (от 10 мин до 24 часов)
- **FR-2.3** Гарантированная доставка: данные не теряются при перезагрузке
- **FR-2.4** Идемпотентная запись: дубликаты не создают новых записей

### 3. Отдача данных

- **FR-3.1** REST API с фильтрацией по устройствам, метрикам, времени
- **FR-3.2** Выдача «сырых» данных и агрегатов (min/max/avg за период)

- **FR-4.3** Настраиваемый cooldown для предотвращения спама

## Нефункциональные требования

### Производительность

- **NFR-1.1** Обработка минимум 1000 сообщений/сек
- **NFR-1.2** Задержка от приёма данных до срабатывания триггера: < 100 мс (внутри локальной сети)
- **NFR-1.3** Буфер 24 часа × 100 устройств × 10 сенсоров ≈ 345 МБ (без сжатия)

## Бизнес-ограничения

1. Система **не управляет** устройствами напрямую (не прошивает, не посылает команд управления)
2. Видео-потоки и тяжёлые бинарные данные **не в скоупе** текущей версии
3. Внешние webhook'и считаются **ненадёжными** — система корректно обрабатывает их недоступность
4. Edge-first: полноценная работа **без постоянного подключения** к облаку
5. **Только on-premise** — данные предприятия не покидают его периметр
6. **Без мультитенантности** — один экземпляр на одно предприятие
7. Интеграция с MES/ERP **не является требованием** текущей версии (будущее расширение)
8. Требований к промышленным сертификациям **нет на текущем этапе**
