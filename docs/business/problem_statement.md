# Проблемное заявление

> **Назначение документа:** зафиксировать, какую боль клиента решает Gateway и чем это ценно для бизнеса. Используется в продажах, питчах, обосновании фаз дорожной карты.

---

## 1. Контекст клиента

Типичный клиент Gateway — **промышленное предприятие с цехом, где параллельно работает оборудование разных вендоров и поколений**: старые контроллеры по Modbus RTU, современные ПЛК по Modbus TCP, единичные устройства с MQTT или HTTP-выходами, периодически — самодельные Arduino-узлы для точечных задач.

Этому предприятию нужно:

1. **Видеть** все данные с оборудования на одном дашборде (температура, давление, счётчики, события).
2. **Реагировать** на аномалии (тревога в Telegram/email/push, автоматическая остановка линии).
3. **Хранить** историю для анализа и отчётов.

---

## 2. Как эта задача решается сегодня (до Gateway)

| Подход | Что делает клиент | Боль |
|--------|-------------------|------|
| **Кастомная интеграция под заказ** | Нанимает подрядчика, который пишет скрипты «под каждый датчик» | Долго, дорого, непереносимо между проектами |
| **Коробочная SCADA (Ignition, WinCC, Кросс)** | Покупает проприетарное решение + инженера на обслуживание | Цена, вендор-лок, сложно расширять, требует лицензий |
| **Сборка на Node-RED / ThingsBoard руками** | Инженер предприятия сам настраивает open-source стек | Нет единой точки ответственности, надёжность «как повезёт», нечему дать в поддержку |
| **Ничего** | Люди ходят и снимают показания руками | Потерянные данные, поздняя реакция на аварии, человеческий фактор |

## 3. Ценностное предложение Gateway

**Gateway превращает подключение нового оборудования из проекта в конфигурацию.**

| Было | Стало с Gateway |
|------|-----------------|
| Подрядчик пишет код под каждый датчик | Инженер АСУ ТП настраивает в Node-RED за полчаса |
| Разнородные данные в разных местах | Единый API и формат для всех протоколов |
| Данные теряются при падении сети | Store-and-forward: 24 часа буфера локально |
| Триггеры и тревоги программирует разработчик | Инженер-технолог создаёт правила визуально |
| Платформу дашбордов интегрирует IT-отдел | Готовый API — подключается «из коробки» |

### Три принципа, которые продаём

1. **«Подключение без кода»** — новое устройство = конфигурация в Node-RED, не релиз ПО.
2. **«Edge-first»** — завод работает, даже когда сеть в облако упала.
3. **«Железо + софт в сборе»** — один поставщик, одна ответственность, одна точка поддержки.

## 4. Что Gateway **не** решает

Чётко очерченные границы помогают не продавать лишнего:

- Не управляет оборудованием (не посылает команды на остановку/запуск).
- Не заменяет MES/ERP — только поставляет им данные.
- Не анализирует видео, не обрабатывает тяжёлые бинарные потоки.
- Не работает как SaaS — только on-premise, в периметре предприятия.
- Не поддерживает мультитенантность — один экземпляр = одно предприятие.

См. также полный список ограничений: [requirements.md → Бизнес-ограничения](requirements.md#бизнес-ограничения).

## 6. Риски и допущения

| Риск | Вероятность | Смягчение |
|------|-------------|-----------|
| Клиент не видит разницы между Gateway и «просто Node-RED» | Высокая | Продавать именно бандл «железо + поддержка», не софт |
| Подрядчики-конкуренты демпингуют ценой разовой интеграции | Средняя | Позиционирование через TCO и расширяемость, не через стартовую цену |
| Первый клиент не масштабируется: специфика отрасли делает решение нетипичным | Средняя | Выбирать первого клиента с типовым стеком (Modbus + стандартные датчики) |
| Отсутствие промышленных сертификаций блокирует крупных клиентов | Низкая (сейчас) | Зафиксировано в [бизнес-ограничениях](requirements.md#бизнес-ограничения); планируется по мере роста |
