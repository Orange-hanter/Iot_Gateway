# Персоны и участники сделки

> **Назначение документа:** различить **покупательские роли** (кто принимает решение о покупке) и **операционные роли** (кто пользуется продуктом каждый день). В [requirements.md](requirements.md#целевые-пользователи) описаны только вторые — здесь дополняем первыми.

---

## Карта участников сделки (B2B)

```
┌──────────────────────────────────────────────────────┐
│ Экономический покупатель ── одобряет бюджет          │
│     ▲                                                 │
│     │ согласование                                    │
│ Лицо, принимающее решение ── выбирает поставщика     │
│     ▲                                                 │
│     │ рекомендация                                    │
│ Технический чемпион ── проверяет и защищает выбор    │
│     ▲                                                 │
│     │ пилот, демо                                     │
│ Операционные пользователи ── работают ежедневно      │
└──────────────────────────────────────────────────────┘
```

В небольшом предприятии несколько ролей может совмещать один человек. В крупном — каждая роль отдельный сотрудник с разными интересами.

## 1. Покупательские роли

### 1.1. Экономический покупатель

**Кто:** Собственник / генеральный директор / директор по производству.

**Что для него важно:**
- Возврат инвестиций: «когда окупится и за счёт чего».
- Предсказуемость: фикс-бюджет на проект, без открытых счётчиков.
- Риски: «не встанет ли цех, если это сломается».

**Как с ним говорить:**
- Через TCO: сравнение с кастомной интеграцией и коробочной SCADA на горизонте 3 лет.
- Через прецеденты: кейс первого клиента с цифрами.
- Через ответственность: «один поставщик за железо, софт и поддержку».

**Возражения, которые он принесёт:**
- «У нас уже есть подрядчик, который это делает дешевле» → ответ через TCO и расширяемость.
- «А если вы исчезнете через год?» → код у клиента (on-premise), open-source компоненты, документация.

### 1.2. Лицо, принимающее решение (ЛПР)

**Кто:** Главный инженер / главный технолог / начальник АСУ ТП.

**Что для него важно:**
- Минимизация простоев оборудования.
- Рост наблюдаемости процесса (раньше увидел — быстрее починил).
- Решение проблемы «зоопарка» из разнородного оборудования.

**Как с ним говорить:**
- Через сценарии: «вот как ваш инженер подключит новый датчик за 30 минут».
- Через надёжность: store-and-forward, watchdog, health checks.
- Через совместимость: список поддерживаемых протоколов.

**Возражения, которые он принесёт:**
- «А наше старое оборудование по Modbus RTU заработает?» → да, через конвертер RS485→Ethernet ([procurement_guide.md](procurement_guide.md)).
- «А если вы не успеваете — кто нам поможет?» → модель поддержки ([support_model.md](support_model.md)).

### 1.3. Технический чемпион

**Кто:** Ведущий инженер АСУ ТП / IT-директор среднего предприятия.

**Что для него важно:**
- Открытость архитектуры: «я должен понять, как это устроено».
- Контроль: «я хочу иметь доступ к данным и коду».
- Репутация: его рекомендация — его лицо.

**Как с ним говорить:**
- Показать [architecture/overview.md](../architecture/overview.md) и [ADR](../decisions/index.md).
- Дать развернуть пилот у себя.
- Признать ограничения честно (что продукт **не** делает — см. [problem_statement.md](problem_statement.md#4-что-gateway-не-решает)).

**Возражения, которые он принесёт:**
- «Почему не чистый Node-RED?» → ответ в `competitive_analysis.md` (будет в Модуле 3).
- «Как выглядит обновление?» → blue-green deployment (Фаза 6 дорожной карты).

## 2. Операционные роли

Сведены в [requirements.md → Целевые пользователи](requirements.md#целевые-пользователи). Ниже — дополнения по болям, которые не влезли в общую таблицу.

### 2.1. Инженер АСУ ТП

- **День без Gateway:** пишет скрипты, отлаживает парсинг пакетов, ругается с вендорами из-за недокументированных регистров.
- **День с Gateway:** добавляет устройство через UI, маппит теги, проверяет на истории, деплоит без рестарта.
- **Метрика успеха:** время на подключение нового устройства. Цель — см. [requirements.md → Метрики успеха](requirements.md#метрики-успеха).

### 2.2. Инженер-технолог

- **День без Gateway:** звонит программисту, чтобы тот «добавил условие в скрипт» для новой тревоги.
- **День с Gateway:** в Node-RED кликает правило «если давление > X более 10 секунд — webhook в Telegram».
- **Метрика успеха:** количество правил, которые технолог создал сам, без программиста.

### 2.3. Оператор / Диспетчер

- **День без Gateway:** смотрит в Excel или на панель ПЛК, звонит сменному мастеру при аварии.
- **День с Gateway:** получает push-уведомление на телефон за секунды, видит дашборд в реальном времени.
- **Метрика успеха:** время от аномалии до уведомления.

### 2.4. Разработчик интеграций (платформа дашбордов)

- **День без Gateway:** пишет парсеры под каждый протокол, поддерживает их.
- **День с Gateway:** один REST API + WebSocket (Фаза 4), единый формат данных.
- **Метрика успеха:** объём кода интеграционного слоя дашборда.

### 2.5. Системный администратор

- **День без Gateway:** непонятно, что мониторить — зоопарк скриптов.
- **День с Gateway:** Prometheus-метрики, health checks, docker-compose, стандартный бэкап БД.
- **Метрика успеха:** MTTR при инцидентах.

## 3. Типовой путь от лида до сделки

<!-- @TODO: валидировать после первых 3–5 сделок и уточнить длительности этапов. -->

| Этап | Кто участвует | Что показываем | Типовая длительность |
|------|---------------|----------------|----------------------|
| Контакт | ЛПР + технический чемпион | Питч, 1-page summary | 1–2 недели |
| Технический просмотр | Чемпион + инженер АСУ ТП | [architecture/overview.md](../architecture/overview.md), демо-стенд | 1–2 недели |
| Пилот на 5–10 сенсорах | Инженер АСУ ТП + оператор | Рабочий стенд, данные в дашборде | 2–4 недели |
| Коммерческое предложение | Экономический покупатель | Спецификация + цена (см. [pricing_model.md](pricing_model.md)) | 1–3 недели |
| Закупка и монтаж | Инженер АСУ ТП | [procurement_guide.md](procurement_guide.md) | 4–6 недель (логистика) |
| Запуск в продуктив | Все | [guides/deployment.md](../guides/deployment.md) | 1–2 недели |

> Ориентировочный цикл сделки: **~3–4 месяца** от первого контакта до запуска в эксплуатацию.
