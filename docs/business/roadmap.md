# Дорожная карта проекта Gateway

> Оценка: 1 разработчик, 1 спринт = 2 недели, 1 SP ≈ 1.5 дня.  
> Сроки ориентировочные, без жёстких дедлайнов.

---

## Фазы

### Фаза 0: Архитектурные решения ✅ / 🔄

**Длительность:** 2–3 недели  
**Цель:** Закрыть открытые решения до старта разработки.

- [x] Выбор Node-RED для visual config (ADR-007)
- [x] Выбор runtime-плагинов через Python (ADR-006)
- [x] Определение модели поставки (on-premise + железо)
- [ ] Выбор СУБД (ADR-005 — анализ завершён, [нужно решение](../decisions/ADR-005-database.md))

---

### Фаза 1: Пилот ✅ Завершена

**Длительность:** 4 недели (завершена)  
**Цель:** Доказать жизнеспособность архитектуры на реальном оборудовании.

Результаты:
- Модульный монолит на Python/FastAPI
- 3 драйвера (generic_json, arduino_mq2, arduino_button_dht11)
- SQLite + WAL mode хранилище
- Rule Engine с webhook dispatch
- Docker Compose deployment
- Оценка надёжности: 6.5/10 → 8.5/10

---

### Фаза 2: Продуктовое ядро на Go

**Оценка:** ~14 недель (7 спринтов)  
**Старт:** После выбора СУБД (ADR-005)  
**Цель:** Переписать критичные сервисы на Go, подключить продуктовую БД.

| Спринт | Задачи | Недели |
|--------|--------|--------|
| S1–S2 | Go: gateway-core (ingestion + normalization + dispatch) | 1–4 |
| S3 | Go: rule-engine (polling, conditions, cooldown) | 5–6 |
| S4 | Go: api-gateway (REST, auth, rate limit) | 7–8 |
| S5 | Миграция БД из SQLite → выбранная СУБД + Alembic аналог | 9–10 |
| S6 | RBAC (роли, скоупы, токены) | 11–12 |
| S7 | Структурированное логирование, аудит конфигурации | 13–14 |

---

### Фаза 3: Интеграционный слой (Python runtime + Modbus)

**Оценка:** ~10 недель (5 спринтов)  
**Зависимости:** Фаза 2 завершена (Go core API готово)  
**Цель:** Python runtime-сервис для адаптеров, первый клиент.

| Спринт | Задачи | Недели |
|--------|--------|--------|
| S8 | Python adapter-runtime: gRPC-мост Go↔Python | 15–16 |
| S9 | Modbus TCP/RTU адаптер (Python, первый клиент) | 17–18 |
| S10 | Node-RED базовая интеграция + кастомные узлы (subscribe/publish) | 19–20 |
| S11 | Runtime-плагины: горячая загрузка Python-адаптеров | 21–22 |
| S12 | Буферизация Store&Forward в Go + DLQ | 23–24 |

---

### Фаза 4: Платформа дашбордов

**Оценка:** ~8 недель (4 спринта)  
**Зависимости:** Фаза 3 (WebSocket API, стабильный data layer)  
**Цель:** Наша платформа для отображения данных.

| Спринт | Задачи | Недели |
|--------|--------|--------|
| S13 | WebSocket API (Go, подписки для дашбордов) | 25–26 |
| S14 | Prometheus + Grafana мониторинг | 27–28 |
| S15 | Система экспорта данных (CSV, Parquet, расписание) | 29–30 |
| S16 | Dashboard API: Live Cache (Redis), агрегаты | 31–32 |

---

### Фаза 5: Расширенная конфигурация

**Оценка:** ~8 недель (4 спринта)  
**Цель:** Полная Node-RED интеграция, профили устройств, визуальные триггеры.

| Спринт | Задачи | Недели |
|--------|--------|--------|
| S17 | Node-RED: полная палитра узлов (история, события, фильтры) | 33–34 |
| S18 | Профили устройств (шаблоны, маппинг, версионирование) | 35–36 |
| S19 | Машина состояний событий + оконная агрегация триггеров | 37–38 |
| S20 | Шифрование секретов, аудит, резервное копирование | 39–40 |

---

### Фаза 6: Масштабирование (будущее)

**Оценка:** ~8+ недель, зависит от потребностей  
**Цель:** Работа на нескольких узлах предприятия.

- OPC UA адаптер
- Kubernetes deployment
- Обновление без простоя (blue-green)
- Multi-node federation
