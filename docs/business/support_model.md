# Модель поддержки и SLA

> **Назначение документа:** описать, что клиент получает после продажи. Какие тарифы есть, как работают уровни L1/L2/L3, за какое время отвечаем на какой инцидент.
>
> **Статус:** черновик. Конкретные значения SLA и канальные детали требуют решения владельца продукта — помечены `@TODO: требует решения`.

---

## 1. Принципы

1. **Одна точка контакта.** Клиент пишет в одно место и получает ответ — даже если проблема в датчике, сети или его собственной инфраструктуре.
2. **Данные и код у клиента.** Доступ к системе клиента — только по его явному согласию на конкретный инцидент.
3. **Эскалация прозрачна.** Если решение требует L2 или L3 — клиент видит, что заявка поднялась, и кто за неё отвечает.
4. **Обновления — не сюрприз.** Плановые релизы с changelog, экстренные патчи согласуются со сменным инженером клиента.

---

## 2. Уровни поддержки

### L1 — Первая линия

- Приём заявок, классификация, проверка типовых проблем (доступ к устройству, настройки, health checks).
- Работает по runbook для стандартных ситуаций.
- Эскалирует на L2, если не решает за <!-- @TODO: требует решения --> минут.

### L2 — Техническая поддержка

- Инженер с доступом к логам и метрикам клиента (по согласованию).
- Разбирает нестандартные ошибки, помогает с Node-RED, настраивает правила вместе с клиентом.
- Эскалирует на L3, если проблема — в ядре продукта.

### L3 — Разработка продукта

- Разработчик Gateway.
- Работает с багами ядра, создаёт хотфиксы, выпускает патч-релизы.
- Контакт с клиентом — только через L2 (не напрямую).

---

## 3. Категории инцидентов

Тяжесть инцидента определяет время реакции, а не канал обращения.

| Категория | Определение | Пример |
|-----------|-------------|--------|
| **P1 — Критическая** | Gateway не работает, данные не принимаются, бизнес-процесс остановлен | Ядро упало и не поднимается, критический триггер не срабатывает |
| **P2 — Высокая** | Работает частично, есть обходной путь | Один протокол отказал, но остальные работают; webhook не доставляется |
| **P3 — Средняя** | Работает, но с неудобствами | Дашборд медленный, логи переполняются |
| **P4 — Низкая / Запрос** | Вопрос, консультация, feature request | Как подключить новый тип датчика, просьба добавить канал уведомлений |

## 5. Обновления и релизный цикл

### Типы релизов

- **Patch (x.y.Z)** — security fixes, исправления критических багов. Выходят по мере готовности, все тарифы.
- **Minor (x.Y.0)** — новый функционал, обратная совместимость сохраняется. Раз в <!-- @TODO --> 1–3 месяца. Basic получает только security-подмножество.
- **Major (X.0.0)** — ломающие изменения, миграция конфигурации. Раз в <!-- @TODO --> 12+ месяцев.

### Процесс накатывания обновлений

1. Уведомление клиента с changelog за <!-- @TODO --> 2–7 дней (в зависимости от уровня).
2. Тестирование на стенде клиента (для Premium — совместное с инженером поддержки).
3. Окно обновления — согласуется с производственным графиком клиента.
4. Возможность отката (docker image предыдущей версии остаётся на узле).

### Политика поддержки версий

- Текущая мажорная версия — полная поддержка.
- Предыдущая мажорная — только security-патчи в течение <!-- @TODO --> 12 месяцев после выхода новой.
- Более старые — не поддерживаются, миграция обязательна.

## 6. Процессы, не относящиеся к инцидентам

### Запросы на доработку (Feature Requests)

- Поступают через обычный канал поддержки с категорией P4.
- L2 передаёт в бэклог ([backlog/user_stories.md](../backlog/user_stories.md)).
- Клиент видит статус своего запроса в тикет-системе.
- Приоритет в бэклоге — на усмотрение product owner'а, но учитывается тариф клиента.

### Обучение

- Включено в единовременные услуги внедрения ([pricing_model.md → 3.1](pricing_model.md#31-единовременная-часть)).
- Дополнительные сессии — платные (см. pricing_model).

## 7. Что **не** входит в поддержку

Важно прописать явно, чтобы избежать спорных случаев:

- Ремонт или замена датчиков и сетевого оборудования — это ответственность клиента.
- Восстановление данных после катастрофического сбоя оборудования, если бэкапы клиента отсутствуют.
- Обучение новых сотрудников клиента помимо стандартного пакета (оплачивается отдельно).
- Разработка кастомных драйверов под уникальное оборудование (оплачивается отдельно, см. pricing_model).
- Доработка сторонних систем, к которым подключается Gateway (MES, ERP, дашборд-платформы).
