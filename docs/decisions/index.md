## ADR-001: SQLite как хранилище для MVP

**Статус:** ✅ Принято  
**Контекст:** Для пилотной версии требовалось лёгкое, встраиваемое хранилище без внешних зависимостей.  
**Решение:** SQLite с WAL mode для параллельного чтения.  
**Последствия:** Ограничение по производительности при >1M событий/день. Запланирована миграция на PostgreSQL.

## ADR-004: Модульный монолит

**Статус:** ✅ Принято  
**Контекст:** ТЗ описывает микросервисную архитектуру, но для MVP это избыточно.  
**Решение:** Модульный монолит с чёткими границами модулей, готовый к декомпозиции.  
**Последствия:** Быстрое прототипирование, единый деплой. При росте нагрузки — выделение сервисов.

## ADR-006: Runtime-плагины через Python adapter-runtime

**Статус:** ✅ Принято (2026-04-08)  
**Контекст:** Нужна возможность добавлять и обновлять адаптеры протоколов без рестарта ядра.  
**Решение:** Отдельный Python-сервис `adapter-runtime` с `importlib.reload()`. Связь с Go-ядром через gRPC.  
**Последствия:** Go-ядро остаётся стабильным; Python берёт на себя динамичность. Требует gRPC контракта.

## ADR-007: Node-RED для device configuration

**Статус:** ✅ Принято (2026-04-08)  
**Контекст:** Нужен визуальный инструмент конфигурирования устройств и потоков данных.  
**Решение:** Node-RED как sidecar в `adapter-runtime`. Custom nodes для publish/subscribe в нашем брокере.  
**Последствия:** Node-RED хранит конфиги потоков. Требует разработки npm-пакета `node-red-contrib-gateway`.

## ADR-008: Go как язык продуктового ядра

**Статус:** ✅ Принято (2026-04-08)  
**Контекст:** Python-монолит не обеспечит 1000 msg/sec при масштабировании. Нужна замена для core-сервисов.  
**Решение:** Go для `gateway-core`, `rule-engine`, `api-gateway`. Python остаётся для `adapter-runtime` (динамическая загрузка) и `export-worker`.  
**Последствия:** Требует миграции с Python (Фаза 2). Разработчики должны знать Go. Производительность: goroutines покрывают 1000 msg/sec с запасом.
