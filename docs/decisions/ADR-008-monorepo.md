# ADR-008: Monorepo с единственным go.mod

> **Статус:** ✅ Принято (20 апреля 2026)  
> **Решение:** Go-код проекта организован как monorepo с единым `go.mod` в корне  
> **Контекст:** Go-ядро растёт с добавлением новых компонентов (API, engine, storage). Нужно выбрать между отдельными модулями (multi-module) или единым monorepo.

---

## Контекст и ограничения

**Требования к структуре Go-проекта:**

| # | Требование | Источник |
|---|-----------|---------|
| R1 | Простота development: один `go.mod`, привычное развитие локально | Команда |
| R2 | Чёткие границы между компонентами (API, engine, storage, drivers) | ADR-004 (модульный монолит) |
| R3 | Возможность внутреннего переиспользования пакетов без git-зависимостей | Dev experience |
| R4 | Единственный binary при сборке — Go-ядро (adapter-runtime отдельный Python) | Архитектура |
| R5 | Версионирование: single version для всех пакетов или гибкость | Развёртывание |
| R6 | CI/CD: тестирование и сборка — один pipeline или на пакет | DevOps |

**Альтернативные подходы:**

- **Multi-module:** отдельный `go.mod` для каждого компонента (api/, engine/, storage/)
- **Multi-repo:** полностью отдельные Git-репозитории для каждого сервиса  
- **Monorepo (выбран):** единый `go.mod`, все пакеты в одном дереве

---

## Рекомендуемая структура (Monorepo)

```
gateway-core/                  ← корень проекта
├── go.mod                     ← единый модуль github.com/your-org/gateway-core
├── go.sum
├── main.go                    ← точка входа (Go-ядро)
├── cmd/                       ← CLI-утилиты и вспомогательные команды
│   └── migrate/
│       └── main.go            ← миграции БД
├── api/                       ← HTTP API, gRPC сервисы
│   ├── http/
│   │   └── handlers.go
│   └── grpc/
│       └── adapter_service.go
├── engine/                    ← ядро обработки (правила, инженерия)
│   ├── rules/
│   │   ├── engine.go
│   │   └── executor.go
│   └── pipeline/
│       └── processor.go
├── storage/                   ← слой данных (CRUD, миграции)
│   ├── database/
│   │   ├── connection.go
│   │   └── models.go
│   └── timescale/
│       ├── hypertables.go
│       └── retention.go
├── drivers/                   ← интеграции с железом (Arduino, MQTT, и т.д.)
│   ├── mqtt/
│   └── arduino/
├── config/                    ← конфигурация
│   └── config.go
├── internal/                  ← приватные утилиты
│   ├── logger/
│   ├── metrics/
│   └── errors.go
└── tests/                     ← интеграционные тесты
    └── integration_test.go
```

**Ключевые правила:**

1. **Один `go.mod`** в корне: `github.com/your-org/gateway-core`
2. **Внутри — пакеты**: импортируются как `github.com/your-org/gateway-core/api`, `github.com/your-org/gateway-core/engine`
3. **Без `go.mod` в подпапках** (исключение: инструменты в `cmd/*` могут быть отдельными, но опционально)
4. **Версионирование:** единая версия для всего проекта (например, v1.0.0 = все пакеты v1.0.0)

---

## Плюсы Monorepo подхода

✅ **Development:**
- Один `go get -u` обновляет всё
- Локальные импорты без Git-зависимостей: `import "github.com/your-org/gateway-core/api"`
- Можно легко переписать внутренний интерфейс между пакетами

✅ **Производительность:**
- Один бинарь → одно развёртывание
- Нет сетевых задержек между микросервисами (пока монолит)
- Единый процесс — простое управление ресурсами

✅ **Тестирование:**
- Один test suite — покрыть всё в одном `go test ./...`
- Легко писать интеграционные тесты между пакетами
- Одна конфигурация CI/CD pipeline

✅ **Рефакторинг:**
- Можно менять интерфейсы между пакетами без git-тегирования
- Go compiler проверит все места использования
- IDE refactoring работает across всеми пакетами

## Минусы Monorepo подхода

❌ **Масштабируемость командой:**
- При 20+ разработчиков могут быть конфликты merge
- Каждый коммит влияет на весь проект
- (Рискуется: рано для этого проекта)

❌ **Нарушение границ:**
- Если нет дисциплины, пакеты могут запутаться в циклических зависимостях
- Нужны правила: `api` не может импортировать `cmd`, только `api` → `engine` → `storage`

❌ **Отдельные версии:**
- Сложнее поддерживать, если `api` нужна v1.2, а `engine` — v1.5
- (Решение: если нужно — переходить на multi-module)

## Архитектурные слои (не нарушать!)

```
Зависимости между пакетами:

┌─────────────────────────────────────────────┐
│ main.go (entrypoint)                       │ ← импортирует api, engine, storage
├─────────────────────────────────────────────┤
│ api/ (HTTP, gRPC)                          │ ← зависит от engine, config
├─────────────────────────────────────────────┤
│ engine/ (бизнес-логика)                    │ ← зависит от storage, drivers, config
├─────────────────────────────────────────────┤
│ storage/ (CRUD, миграции)                  │ ← зависит только от config, internal
├─────────────────────────────────────────────┤
│ drivers/ (интеграции)                      │ ← зависит от config, internal
├─────────────────────────────────────────────┤
│ config/, internal/ (утилиты)               │ ← не зависят ни от чего
└─────────────────────────────────────────────┘
```

**Правило циклических зависимостей:** запрещены!
- ❌ `api → engine → api` (циклическая зависимость)
- ✅ `api → engine → storage` (DAG)

Go compiler будет жаловаться на циклические импорты при компиляции.

## Миграция на Multi-Module (если понадобится)

Если позже понадобится отдельное версионирование или отделение сервиса:

```bash
# Перейти от monorepo к multi-module:
mkdir -p storage
cp -r storage/* storage/  # переместить пакет
cd storage
go mod init github.com/your-org/gateway-storage
go mod edit -require=github.com/your-org/gateway-core@v1.0.0  # новая зависимость
```

Это обратимо и Go позволяет такие переходы без потери истории.
