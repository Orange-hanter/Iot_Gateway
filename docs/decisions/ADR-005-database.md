# ADR-005: Выбор СУБД для продуктовой версии

> **Статус:** ✅ Принято (20 апреля 2026)  
> **Решение:** TimescaleDB как основная СУБД  
> **Контекст:** Проект переходит с SQLite (MVP) на продуктовую СУБД. Выбор влияет на ORM в Go, производительность на 1000 msg/sec, и долгосрочные возможности аналитики.

---

## Контекст и ограничения

**Требования, влияющие на выбор:**

| # | Требование | Источник |
|---|-----------|---------|
| R1 | 1000 msg/sec запись при 100 устройствах | NFR-2.1 |
| R2 | On-premise развёртывание на ПК/NUC | Бизнес-ограничение |
| R3 | Хранение временных рядов + реляционных данных (правила, устройства) | FR-2 + FR-5 |
| R4 | Миграция с текущей SQLAlchemy/SQLite схемы | Техническое ограничение |
| R5 | Простота обслуживания для non-DBA команды | Операционное ограничение |
| R6 | Лицензия: open source или permissive | Бизнес-ограничение |

**Примерный объём данных:**
- 100 устройств × 10 msg/sec × 86400 сек/день = **86.4M событий/день**
- При 50 байт/событие: ~4.3 GB/день сырых данных
- С компрессией: ~0.4–0.8 GB/день
- Retention: 90 дней → ~40–70 GB на диске

---

## Рассматриваемые варианты

### Вариант A: TimescaleDB (PostgreSQL extension)

**Что это:** Расширение PostgreSQL, превращающее его в автоматически шардированную time-series БД с сохранением полного SQL.

**Ключевые возможности:**
- `hypertable` — автоматическое разбиение на чанки (по времени)
- `continuous aggregates` — материализованные агрегаты, автообновляются
- `compression` — до 95% сжатие для старых чанков
- `data retention policies` — автоудаление по расписанию
- Полный PostgreSQL SQL — JOIN, JSONB, FOR UPDATE, etc.

**Производительность:**
- Запись: ~200K-500K строк/сек (сырые вставки, ограничено диском)
- 1000 msg/sec = 1000 rows/sec — **с огромным запасом**
- Range queries по времени: быстро (индекс по `created_at` в каждом чанке)

**Pros:**
- ✅ Совместимость с PostgreSQL: привычный SQL, любые ORM
- ✅ Легкая миграция с SQLite через Alembic (те же модели SQLAlchemy)
- ✅ `continuous aggregates` закрывают NFR-1.3 (агрегация последнего часа)
- ✅ JSONB для гибких payload-полей
- ✅ Активное open-source сообщество, Apache License (Community edition)
- ✅ Docker образ: `timescale/timescaledb:latest-pg16`

**Cons:**
- ❌ Нужно понимать PostgreSQL администрирование (туning, backup, pg_dump)
- ❌ Тяжелее SQLite (~150MB RAM минимум)
- ❌ TimescaleDB community edition = ограниченные enterprise-фичи (multi-node в платной)

**Оценка требований:**
| R | Статус |
|---|--------|
| R1 | ✅ Легко |
| R2 | ✅ Docker on-premise |
| R3 | ✅ Hypertable + реляционные таблицы |
| R4 | ✅ SQLAlchemy поддерживает pg без изменений |
| R5 | 🟡 Требует базовых знаний PostgreSQL |
| R6 | ✅ Apache 2.0 (community) |

---

### Вариант B: PostgreSQL (vanilla)

**Что это:** Классический PostgreSQL без time-series расширений.

**Pros:**
- ✅ Наиболее распространённая реляционная СУБД — максимальная экспертиза
- ✅ Полная совместимость с SQLAlchemy — миграция тривиальна
- ✅ JSONB, CTE, window functions, RBAC встроены
- ✅ Лучший выбор для конфигурационных данных (устройства, правила)

**Cons:**
- ❌ Нет встроенной time-series оптимизации — range queries медленны при >100M строк
- ❌ Нужно вручную создавать партиции (pg_partman extension) для временных рядов
- ❌ Нет нативной компрессии для старых данных
- ❌ Нет `continuous aggregates` — придётся писать материализованные views вручную

**Оценка требований:**
| R | Статус |
|---|--------|
| R1 | ✅ Справляется на старте, деградирует при >100M строк без партиций |
| R2 | ✅ Docker on-premise |
| R3 | 🟡 Hypertables нет, ручные партиции |
| R4 | ✅ Прямая миграция |
| R5 | 🟡 Базовые знания PostgreSQL |
| R6 | ✅ PostgreSQL License |

---

### Вариант C: InfluxDB v2 (OSS)

**Что это:** Purpose-built time-series база данных с собственным языком запросов Flux.

**Pros:**
- ✅ Лучшая производительность для write-heavy IoT workload
- ✅ Retention policies, downsampling — встроены
- ✅ Grafana интеграция — нативная
- ✅ Лёгкий Docker образ

**Cons:**
- ❌ **Flux вместо SQL** — крутая кривая обучения, нет поддержки в стандартных ORM
- ❌ Не подходит для хранения конфигурации (правила, устройства) — нужно 2 БД
- ❌ InfluxDB v3 сменила лицензию на BSL (Business Source License) — не open-source
- ❌ v2 (MIT) уже не активно разрабатывается; v3 — proprietary с ограничениями
- ❌ Нет JOIN — реляционные запросы невозможны

**Оценка требований:**
| R | Статус |
|---|--------|
| R1 | ✅ High write throughput |
| R2 | ✅ Docker |
| R3 | ❌ Только time-series; нужна вторая БД для конфигурации |
| R4 | ❌ Совместимости с SQLAlchemy нет |
| R5 | ❌ Flux язык — отдельное обучение |
| R6 | ⚠️ v2 MIT, v3 BSL — рискованно |

---

### Вариант D: QuestDB

**Что это:** Высокопроизводительная time-series БД с PostgreSQL wire protocol (частично).

**Pros:**
- ✅ Экстремально высокая write throughput (млн строк/сек)
- ✅ SQL-совместимый интерфейс (PostgreSQL wire protocol)
- ✅ Лёгкий и быстрый

**Cons:**
- ❌ Молодой проект — меньше документации, edge cases
- ❌ Ограниченная поддержка DML (UPDATE/DELETE ограничены)
- ❌ Нет нативного JSONB
- ❌ ORM поддержка через PostgreSQL wire — может быть нестабильной
- ❌ Меньшее сообщество, меньше операционного опыта

**Оценка требований:**
| R | Статус |
|---|--------|
| R1 | ✅ Избыточно мощная |
| R2 | ✅ Docker |
| R3 | 🟡 Time-series хорошо; реляционные — ограниченно |
| R4 | 🟡 PostgreSQL wire, но нет полного SQLAlchemy mapping |
| R5 | 🟡 Менее известна, меньше экспертизы |
| R6 | ✅ Apache 2.0 |

---

### Вариант E: ClickHouse

**Что это:** Аналитическая OLAP-база данных для больших объёмов данных.

**Pros:**
- ✅ Лучшая производительность для аналитических запросов (агрегации, GROUP BY)
- ✅ Превосходное сжатие (10-100x)
- ✅ SQL-совместимый

**Cons:**
- ❌ **OLAP-ориентированная** — не OLTP: UPDATE/DELETE неэффективны, транзакций нет
- ❌ Batch insert only — нельзя вставлять по 1 строке (нужен буфер)
- ❌ Не подходит для хранения правил/конфигурации (нужна вторая БД)
- ❌ Оверхед для 100 устройств/1000 msg/sec — избыточна

**Оценка требований:**
| R | Статус |
|---|--------|
| R1 | ✅ Если с буфером |
| R2 | ✅ Docker |
| R3 | ❌ Только аналитика; нужна вторая БД |
| R4 | ❌ Нет SQLAlchemy совместимости |
| R5 | ❌ Сложна в настройке |
| R6 | ✅ Apache 2.0 |

## Рекомендация

> **Рекомендуется: TimescaleDB**

**Обоснование:**
1. Покрывает все требования (R1–R6) без компромиссов
2. Миграция с SQLite через Alembic — минимальные изменения кода (`SERIAL` → `BIGSERIAL`, добавить `SELECT create_hypertable(...)`)
3. `continuous aggregates` автоматически закрывают NFR-1.3 (последний час агрегаций)
4. `compression policy` решает 90-дневный retention без ручных партиций
5. Go: стандартный `pgx` или `sqlc` — полная поддержка PostgreSQL wire
6. Apache 2.0 — нет лицензионных рисков для on-premise продукта

**Запасной вариант если TimescaleDB окажется лишним:** PostgreSQL vanilla с ручным `pg_partman` — всё ещё лучше SQLite, и код менять не нужно.
