# Пользовательские истории проекта
## Система сбора данных для MES/IIoT (Edge-уровень)

> **Формат:** `As a [роль], I want [возможность], so that [ценность]`
> **Приоритеты:** 🔴 Must have | 🟡 Should have | 🟢 Could have | ⚪ Won't have (сейчас)

---

## 📋 Глоссарий ролей

| Роль | Описание |
|------|----------|
| **Инженер АСУ ТП** | Специалист, настраивающий подключение оборудования, маппинг тегов, профили устройств |
| **Инженер-технолог** | Пользователь, описывающий бизнес-правила, триггеры, сценарии оповещений |
| **Оператор / Диспетчер** | Конечный пользователь, наблюдающий за процессом, получающий алерты |
| **Разработчик интеграций** | Программист, подключающий внешние системы (MES, ERP, BI) к ядру |
| **Системный администратор** | Специалист, отвечающий за развёртывание, мониторинг и поддержку инфраструктуры |
| **Руководитель проекта** | Лицо, принимающее решения о приоритетах, бюджете и сроках внедрения |

## 🗂️ Эпик 1: Настройка и конфигурация системы

### US-1.1: Регистрация нового устройства
```
As an Инженер АСУ ТП,
I want добавить новое устройство через конфигурационный файл или интерфейс,
so that система начала опрашивать его и собирать данные без изменения кода ядра.

Acceptance Criteria:
• Устройство описывается в YAML/JSON (id, тип адаптера, параметры подключения)
• После перезагрузки конфига устройство появляется в реестре без рестарта ядра
• При ошибке подключения система логирует причину и пробует переподключиться
• Устройство отображается в списке с статусом: connected/disconnected/error

Priority: 🔴 Must have
Estimate: 5 SP
```

### US-1.2: Создание профиля устройства (шаблон)
```
As an Инженер АСУ ТП,
I want создать профиль устройства с маппингом физических адресов на логические теги,
so that повторно использовать конфигурацию для однотипного оборудования.

Acceptance Criteria:
• Профиль содержит: имя, версию, список тегов с правилами трансформации
• Профиль применяется к любому устройству соответствующего типа
• При обновлении профиля можно выбрать: применить ко всем или только к новым устройствам
• Система ведёт историю версий профилей

Priority: 🔴 Must have
Estimate: 8 SP
```

### US-1.3: Горячее обновление конфигурации
```
As an Системный администратор,
I want изменять конфигурацию системы без полной перезагрузки,
so that не прерывать сбор данных с работающих линий.

Acceptance Criteria:
• Изменение конфига тегается и валидируется перед применением
• Новые теги начинают опрашиваться, удалённые — останавливаются
• Активные соединения с устройствами не разрываются при обновлении
• В случае ошибки валидации система откатывается к предыдущей версии конфига

Priority: 🟡 Should have
Estimate: 8 SP
```

## 🗂️ Эпик 2: Интеграция устройств и сбор данных

### US-2.1: Подключение устройства по новому протоколу
```
As an Инженер АСУ ТП,
I want добавить поддержку нового протокола (например, OPC UA) через плагин,
so that интегрировать оборудование, не предусмотренное в базовой поставке.

Acceptance Criteria:
• Адаптер реализует стандартный интерфейс (connect, read, write, subscribe)
• Плагин загружается динамически из папки plugins/ без пересборки ядра
• Новый тип адаптера доступен для выбора в конфигурации устройства
• Ошибки в плагине изолированы и не падают всё ядро

Priority: 🟡 Should have
Estimate: 13 SP
```

### US-2.2: Нормализация данных от разнородных устройств
```
As an Разработчик интеграций,
I want получать данные в едином формате независимо от источника,
so что внешние системы не зависели от специфики протоколов.

Acceptance Criteria:
• Все входящие данные приводятся к модели: {device_id, tag, value, timestamp, quality}
• Временные метки конвертируются в UTC с сохранением информации о источнике
• Коды качества (good/uncertain/bad) проставляются автоматически при ошибках чтения
• Типы данных валидируются согласно декларации тега

Priority: 🔴 Must have
Estimate: 5 SP
```

### US-2.3: Обработка сложных схем устройств (битовые маски, составные значения)
```
As an Инженер АСУ ТП,
I want описывать правила преобразования сырых данных в инженерные величины,
so что потребители получали готовые к использованию значения.

Acceptance Criteria:
• В конфиге тега поддерживаются: scale, offset, битовые операции, конвертация типов
• Поддерживается составной маппинг (например, два регистра → один float32)
• Ошибки преобразования не останавливают опрос, а проставляют quality=bad
• Логика преобразования тестируется на симуляторе до применения на реальном устройстве

Priority: 🟡 Should have
Estimate: 8 SP
```

### US-2.4: Буферизация данных при потере связи
```
As an Системный администратор,
I want чтобы данные сохранялись локально при обрыве связи с центром,
so что не терять информацию при сетевых сбоях.

Acceptance Criteria:
• При недоступности верхнего уровня данные накапливаются в локальной очереди
• Объём буфера настраивается (по времени или по количеству записей)
• После восстановления связи данные отправляются пакетом с оригинальными временными метками
• При переполнении буфера применяется настраиваемая стратегия: drop oldest / drop newest / alert

Priority: 🔴 Must have
Estimate: 5 SP
```

## 🗂️ Эпик 3: Обработка событий и триггеры

### US-3.1: Создание правила-триггера с условиями
```
As an Инженер-технолог,
I want настроить правило срабатывания события по условию на теги,
so что система автоматически реагировала на отклонения процесса.

Acceptance Criteria:
• Условие описывается декларативно: "тег оператор значение [логическая связка ...]"
• Поддерживаются временные окна: "условие выполняется более N секунд"
• Правило имеет приоритет (low/medium/high/critical)
• При срабатывании генерируется событие с метаданными: что, когда, значение, контекст

Priority: 🔴 Must have
Estimate: 8 SP
```

### US-3.2: Настройка сценариев оповещений
```
As an Инженер-технолог,
I want привязать к событию один или несколько каналов оповещения,
so что ответственные лица получали информацию нужным способом.

Acceptance Criteria:
• Поддерживаются каналы: email, SMS, webhook, Telegram, локальный алерт
• Для каждого канала настраивается шаблон сообщения с подстановкой переменных
• Можно задать расписание: когда оповещения активны (например, только в смену)
• Подтверждение получения оповещения (ack) меняет статус события

Priority: 🟡 Should have
Estimate: 5 SP
```

### US-3.3: Визуальное программирование логики в Node-RED
```
As an Инженер-технолог,
I want создавать и редактировать сценарии обработки данных через визуальный интерфейс,
so что быстро прототипировать сложную логику без участия разработчиков.

Acceptance Criteria:
• В интерфейсе ядра доступен редактор флоу с ограниченной палитрой узлов
• Предоставлены кастомные узлы: subscribe to tag, write tag, query history, send event
• Флоу можно протестировать на исторических данных перед деплоем
• Деплой флоу применяется "на лету" без остановки сбора данных

Priority: 🟢 Could have
Estimate: 13 SP
```

### US-3.4: Машина состояний для событий
```
As an Оператор,
I want видеть и управлять жизненным циклом события (активно/подтверждено/закрыто),
so что не пропускать критические ситуации и фиксировать реакцию.

Acceptance Criteria:
• Событие имеет статусы: pending → active → acknowledged → resolved
• Оператор может подтвердить событие с комментарием
• Система фиксирует время каждого перехода для отчётности
• Неподтверждённые критические события эскалируются через заданное время

Priority: 🟡 Should have
Estimate: 5 SP
```

## 🗂️ Эпик 4: Хранение и доступ к данным

### US-4.1: Запись и хранение временных рядов
```
As an Разработчик интеграций,
I want чтобы система надёжно сохраняла историю данных с настраиваемыми политиками,
so что можно было строить отчёты и анализировать тренды.

Acceptance Criteria:
• Данные записываются пачками для оптимизации I/O
• Поддерживаются политики ретенции: сырые данные → агрегаты → удаление
• Агрегация (min/max/avg) вычисляется автоматически по расписанию
• Запись идемпотентна: дубликаты по {тег, время} не создают новых записей

Priority: 🔴 Must have
Estimate: 8 SP
```

### US-4.2: API для получения истории данных
```
As an Разработчик интеграций,
I want запрашивать исторические данные через REST API с фильтрацией,
so что внешние системы могли получать нужные данные без прямого доступа к БД.

Acceptance Criteria:
• Поддерживаются фильтры: по тегам, временному диапазону, качеству, интервалу
• Ответ включает метаданные: единицы измерения, источник, статистика выборки
• Поддерживается пагинация и сжатие ответа (gzip)
• API имеет лимиты запросов и аутентификацию по токенам

Priority: 🔴 Must have
Estimate: 5 SP
```

### US-4.3: Подписка на поток данных в реальном времени
```
As an Разработчик интеграций,
I want подписаться на обновления тегов через WebSocket / MQTT,
so что получать данные с минимальной задержкой без постоянного опроса.

Acceptance Criteria:
• Клиент подписывается на один или несколько тегов по паттерну
• Система пушит обновления только при изменении значения или качества
• Поддержка фильтрации: отправлять только если изменение > порога
• При разрыве соединения клиент может запросить пропущенные данные по времени

Priority: 🟡 Should have
Estimate: 8 SP
```

### US-4.4: Метаданные тегов и самодокументирование
```
As an Оператор,
I want видеть описание, единицы измерения и допустимый диапазон для каждого тега,
so что правильно интерпретировать значения на дашборде.

Acceptance Criteria:
• У каждого тега есть атрибуты: displayName, unit, min/max, description, category
• Метаданные доступны через API и отображаются в интерфейсе при наведении
• При изменении метаданных сохраняется история для аудита
• Система может выгрузить документацию по всем тегам в PDF/Excel

Priority: 🟢 Could have
Estimate: 3 SP
```

## 🗂️ Эпик 5: Эксплуатация и мониторинг

### US-5.1: Мониторинг здоровья системы и компонентов
```
As an Системный администратор,
I want видеть метрики работоспособности всех модулей в реальном времени,
so что быстро выявлять и устранять проблемы до влияния на сбор данных.

Acceptance Criteria:
• Каждый компонент экспортирует метрики: статус, загрузка, ошибки, задержки
• Доступен дашборд с обзором: устройства онлайн/оффлайн, очередь записи, место на диске
• При превышении порогов генерируются предупреждения для администратора
• Метрики доступны для интеграции с внешними системами мониторинга (Prometheus, Zabbix)

Priority: 🔴 Must have
Estimate: 5 SP
```

### US-5.2: Логирование с контекстом и трассировкой
```
As an Инженер АСУ ТП,
I want чтобы логи содержали достаточно контекста для диагностики проблем,
so что быстро находить корень сбоя без доступа к исходному коду.

Acceptance Criteria:
• Все логи структурированы (JSON) с обязательными полями: timestamp, level, component, message
• Каждому запросу присваивается correlation_id, который проходит через все слои
• Логи можно фильтровать по устройству, тегу, уровню важности, временному диапазону
• Поддерживается ротация и архивация логов с настраиваемым сроком хранения

Priority: 🟡 Should have
Estimate: 5 SP
```

### US-5.3: Резервное копирование и восстановление конфигурации
```
As an Системный администратор,
I want автоматически создавать бэкапы конфигурации и флоу,
so что быстро восстановить систему после сбоя или ошибки настройки.

Acceptance Criteria:
• Бэкап включает: конфиги устройств, профили, правила триггеров, флоу Node-RED
• Расписание бэкапов настраивается (ежедневно/еженедельно)
• Восстановление из бэкапа возможно через интерфейс или CLI
• Бэкапы шифруются и могут отправляться на внешний носитель / в облако

Priority: 🟡 Should have
Estimate: 5 SP
```

### US-5.4: Обновление ПО без простоя
```
As an Системный администратор,
I want обновлять версию ядра и компонентов без остановки сбора данных,
so что получать исправления и новые функции без влияния на производство.

Acceptance Criteria:
• Поддерживается blue-green или canary-развёртывание компонентов
• Конфигурация и данные сохраняются при обновлении
• В случае проблем возможен быстрый откат к предыдущей версии
• Процесс обновления логируется и отображает прогресс в интерфейсе

Priority: 🟢 Could have
Estimate: 13 SP
```

## 🗂️ Эпик 6: Развёртывание и масштабирование

### US-6.1: Автоматическая инициализация edge-устройства
```
As an Системный администратор,
I want чтобы новое устройство при подключении к сети автоматически получало конфигурацию,
so что минимизировать ручную настройку при масштабировании.

Acceptance Criteria:
• Устройство при первом запуске регистрируется в центральном реестре
• Загружает актуальную конфигурацию из центра (или локальный шаблон)
• Отправляет диагностическую информацию: модель, версия ПО, доступные интерфейсы
• При потере связи с центром работает по последней известной конфигурации

Priority: 🟢 Could have
Estimate: 8 SP
```

### US-6.2: Горизонтальное масштабирование сбора данных
```
As an Руководитель проекта,
I want добавлять новые edge-узлы для распределения нагрузки,
so что система росла вместе с потребностями предприятия.

Acceptance Criteria:
• Несколько узлов могут собирать данные с разных групп устройств
• Центральный компонент агрегирует данные и метаданные со всех узлов
• Конфигурация синхронизируется между узлами (или управляется централизованно)
• При отказе одного узла его нагрузка может быть перераспределена

Priority: ⚪ Won't have (сейчас)
Estimate: 20 SP
```

### US-6.3: Проверка соответствия аппаратным требованиям
```
As an Системный администратор,
I want чтобы система при старте проверяла ресурсы и предупреждала о нехватке,
so что избегать деградации производительности из-за неподходящего железа.

Acceptance Criteria:
• При запуске проверяется: свободное место на диске, доступная ОЗУ, температура ЦП
• Если ресурсы ниже порога — генерируется предупреждение в лог и интерфейс
• Система предлагает рекомендации: какие политики хранения изменить, что отключить
• Метрики использования ресурсов доступны для мониторинга в реальном времени

Priority: 🟢 Could have
Estimate: 3 SP
```

## 🗂️ Эпик 7: Безопасность и аудит

### US-7.1: Аутентификация и авторизация доступа к API
```
As an Разработчик интеграций,
I want получать доступ к данным только после аутентификации и с нужными правами,
so что обеспечить безопасность данных и соответствие требованиям предприятия.

Acceptance Criteria:
• Поддерживается аутентификация по API-токенам с настраиваемым сроком жизни
• Токены имеют скоупы: read-only, write, admin, доступ к конкретным тегам
• Неавторизованные запросы логируются и возвращают 401/403
• Поддержка ротации токенов без простоя клиентов

Priority: 🔴 Must have
Estimate: 5 SP
```

### US-7.2: Аудит изменений конфигурации и действий пользователей
```
As an Руководитель проекта,
I want фиксировать кто, когда и какие изменения внёс в систему,
so что обеспечивать подотчётность и расследовать инциденты.

Acceptance Criteria:
• Все изменения конфигурации, деплои флоу, изменения прав логируются
• В логе: пользователь, время, действие, старые/новые значения, причина (комментарий)
• Логи аудита защищены от модификации и доступны только администраторам
• Поддержка экспорта аудита для внешнего анализа

Priority: 🟡 Should have
Estimate: 5 SP
```

### US-7.3: Шифрование чувствительных данных
```
As an Системный администратор,
I want чтобы учётные данные и критичные данные хранились в зашифрованном виде,
so что соответствовать требованиям безопасности и защищаться от утечек.

Acceptance Criteria:
• Пароли, токены, сертификаты в конфигурации шифруются (AES-256)
• Ключи шифрования хранятся отдельно от данных (TPM / HSM / env vars)
• При передаче по сети используется TLS 1.2+
• Система поддерживает ротацию ключей шифрования

Priority: 🟡 Should have
Estimate: 8 SP
```

## 💡 Примечания для команды

1. **Оценка сложности (SP):** Указана в условных "story points". 1 SP ≈ 0.5-1 день работы одного разработчика.
2. **Зависимости:** Некоторые истории (например, US-3.3 Node-RED) требуют готовности базового API (US-4.2) и системы плагинов (US-2.1).
3. **Тестирование:** Для каждой истории должны быть определены:
   - Юнит-тесты логики
   - Интеграционные тесты с симулятором устройств
   - Приёмочные тесты с реальным сценарием пользователя
4. **Документация:** Параллельно с разработкой ведётся документация конфигурации и API — это часть Definition of Done.

> ✅ **Критерий готовности эпика:** Все 🔴 Must have истории эпика реализованы, протестированы и задокументированы.
