# 📘 АРХИТЕКТУРНАЯ КОНЦЕПЦИЯ
## Модуль сбора данных для MES/IIoT (Edge-уровень)

**Версия:** 1.0  
**Дата:** Апрель 2026  
**Статус:** На согласовании  
**Автор:** Архитектурная рабочая группа  
**Область:** Концептуальное проектирование, уровень абстракции: модули и потоки

---

## 1. Введение

### 1.1. Цель документа
Определить фундаментальные архитектурные решения, границы модулей, принципы взаимодействия и нефункциональные требования к Edge-модулю сбора данных. Документ служит отправной точкой для детального проектирования, разработки и пилотного внедрения.

### 1.2. Область применения
Концепция охватывает:
- Локальный сбор, нормализацию и маршрутизацию данных с разнородного промышленного оборудования
- Обработку событий и триггеров на edge-уровне
- Визуальное конфигурирование логики и экспорт данных
- Обеспечение надёжности, безопасности и наблюдаемости

*Не охватывает:* выбор конкретных технологий, написание кода, скрипты развёртывания, интеграцию со сторонними SaaS-платформами.

### 1.3. Термины и определения
| Термин | Определение |
|--------|-------------|
| **Edge-устройство** | Локальный вычислительный узел, разворачиваемый в цехе/на линии |
| **Адаптер** | Изолированный компонент, реализующий протокол конкретного устройства |
| **Физический тег** | Адрес регистра, бит, путь или идентификатор в устройстве |
| **Логический тег** | Семантическое имя данных в предметной области, стабильное во времени |
| **Ядро (Core)** | Центральный модуль нормализации, маршрутизации и управления состоянием |
| **Unified Namespace** | Единое иерархическое пространство имён для потребителей данных |
| **Профиль устройства** | Декларативный шаблон маппинга физика → логика + правила трансформации |

## 2. Видение и задачи системы

### 2.1. Видение
Создать универсальный, конфигурируемый и отказоустойчивый модуль сбора данных, который становится «единым языком» между оборудованием цеха и бизнес-системами предприятия.

### 2.2. Ключевые задачи
- ✅ Абстрагировать бизнес-логику от физической сложности устройств
- ✅ Обеспечить гарантированную доставку данных при сетевых сбоях
- ✅ Предоставить стабильный API и потоки реального времени для MES/BI
- ✅ Позволить инженерам настраивать правила, алерты и сценарии без изменения кода
- ✅ Сохранять работоспособность в автономном режиме (offline-capable)

---

## 3. Ключевые архитектурные принципы

| Принцип | Суть | Обоснование |
|---------|------|-------------|
| **Семантическая абстракция** | Разделение физических адресов и логических имён | Замена оборудования или обновление прошивки не ломает верхнеуровневую логику |
| **Модульность через контракты** | Чёткие интерфейсы между слоями, плагин-модель адаптеров | Независимая разработка, тестирование и замена компонентов |
| **Асинхронный конвейер** | Разделение потоков сбора, хранения и обработки событий | Сбой в одном потоке не блокирует остальные, предсказуемая задержка |
| **Конфигурация > Код** | Декларативные профили, YAML/JSON, визуальные редакторы | Быстрая интеграция, снижение порога входа, горячее обновление |
| **Edge-First** | Полноценная работа локально, облако — опциональный потребитель | Независимость от WAN, соответствие требованиям промышленности |
| **Наблюдаемость по умолчанию** | Метрики, структурные логи, трассировка запросов | Быстрая диагностика, прогнозирование сбоев, аудит |
| **Идемпотентность и качество данных** | Защита от дубликатов, коды качества `good/uncertain/bad` | Достоверность аналитики, корректная работа триггеров |

### 4.1. Границы модулей
- **Интеграционный слой:** знает всё о протоколах, адресах, байтовых масках. Ничего не знает о бизнес-правилах.
- **Ядро:** знает всё о логических тегах, состояниях, правилах, маршрутизации. Ничего не знает о Modbus/OPC UA/HTTP.
- **API/Экспорт:** знает всё о потребителях, форматах, квотах, расписаниях. Не имеет прямого доступа к адаптерам.
- **Хранилище:** оптимизировано под временные ряды, агрегаты и политики удаления. Не содержит бизнес-логики.

## 5. Модель потоков данных

1. **Приём:** Адаптер опрашивает устройство или получает push-уведомление → преобразует в `{device_id, logical_tag, value, timestamp, quality}` → отправляет в шлюз нормализации.
2. **Маршрутизация:** Ядро копирует сообщение в три независимых потока:
   - 📥 **Хранение:** пакетная запись в time-series, обновление агрегатов
   - ⚡ **Триггеры:** проверка условий, оконная агрегация, генерация событий
   - 💾 **Live Cache:** обновление последнего значения для мгновенного API-ответа
3. **Обработка событий:** Rule Engine управляет состояниями (`pending → active → acknowledged → resolved`). События маршрутизируются в подсистему оповещений и в API.
4. **Экспорт:** Внешние системы запрашивают историю или подписываются на обновления. Система применяет фильтры, квоты, пагинацию и возвращает результат в запрошенном формате.
5. **Визуальная логика (опционально):** Node-RED получает поток данных через кастомные узлы, исполняет сценарий в песочнице, возвращает команды/события обратно в Ядро.

## 6. Стратегия интеграции и расширяемости

### 6.1. Адаптеры как плагины
- Реализуют единый контракт: `connect()`, `read(tags)`, `write(tag, value)`, `subscribe(callback)`, `getDiagnostics()`
- Загружаются динамически из каталога `plugins/` без перезапуска ядра
- Изолированы: падение одного адаптера не влияет на остальные

### 6.2. Профили устройств и семантический маппинг
- Физическая карта регистров описывается в YAML-профиле
- Профиль связывает физические адреса с логическими тегами, задаёт формулы, битовые маски, диапазоны
- Версионирование профилей позволяет мигрировать оборудование без изменения конфигурации потребителей

### 6.3. Визуальный конфигуратор (Node-RED)
- Запускается как изолированный sidecar-процесс
- Связь с ядром через защищённый локальный канал
- Ограниченная палитра узлов, валидация флоу перед деплоем, контекст хранится в основном хранилище
- Позволяет технологам создавать сложные сценарии обработки без участия разработчиков

## 7. Принципы надёжности и отказоустойчивости

| Механизм | Назначение |
|----------|------------|
| **Store & Forward** | Локальный буфер при потере связи с верхним уровнем или хранилищем |
| **Write-Ahead Log** | Запись входящих данных в журнал до обработки → восстановление после сбоя |
| **Graceful Degradation** | При отказе хранилища: сбор и триггеры работают, запись откладывается |
| **Health Checks & Retry** | Экспоненциальная задержка переподключения, автоматический перезапуск зависших адаптеров |
| **Идемпотентная запись** | Ключ `{logical_tag, timestamp}` предотвращает дубликаты при повторной доставке |
| **Контекстная персистентность** | Счётчики, состояния правил и переменные сценариев сохраняются между перезапусками |

## 8. Безопасность и контроль доступа

- **Принцип минимальных привилегий:** адаптеры читают только явно разрешённые теги, пользователи API — только назначенные скоупы
- **Изоляция секретов:** пароли, токены, сертификаты хранятся в зашифрованном виде, доступ через защищённый менеджер
- **Валидация на границе:** все входящие данные проходят проверку типа, диапазона, частоты до попадания в ядро
- **Песочница логики:** выполнение пользовательских скриптов ограничено по времени, памяти и доступу к ресурсам
- **Аудит:** все изменения конфигурации, деплои, доступ к данным логируются с указанием субъекта, времени и контекста
- **Сетевая изоляция:** внешние интерфейсы доступны только через авторизованный шлюз, внутреннее взаимодействие — через localhost/IPC

## 9. Развёртывание и масштабирование

### 9.1. Базовая конфигурация (Single-Node Edge)
Все компоненты работают на одном устройстве. Рекомендуемые ресурсы: 4 ядра, 8 ГБ ОЗУ, 64–128 ГБ industrial SSD, 2× GbE, температурный диапазон –20…+60°C.

### 9.2. Стратегия масштабирования
| Сценарий | Подход |
|----------|--------|
| Рост нагрузки на одной линии | Вертикальный апгрейд (CPU/RAM/SSD) |
| Добавление новых цехов/линий | Горизонтальное развёртывание дополнительных edge-узлов |
| Централизация аналитики | Federation: edge-узлы агрегируют данные и передают в центральный сервер/облако |
| Высокая доступность | Резервный узел с синхронизацией конфигурации и горячей заменой |

### 9.3. Обновления и бэкапы
- Конфигурация и флоу версионируются, поддерживается hot-reload и автоматический rollback при ошибках валидации
- Резервное копирование включает: профили, правила триггеров, визуальные сценарии, метаданные тегов
- Обновление ПО поддерживается по схеме blue-green или canary без остановки сбора данных

## 11. Заключение и следующие шаги

Архитектурная концепция определяет модуль сбора данных как **независимый, конфигурируемый и наблюдаемый edge-компонент**, способный интегрироваться в любое предприятие без привязки к конкретным технологиям или типам оборудования. Разделение физических и логических сущностей, асинхронный конвейер и изоляция сбоев обеспечивают устойчивость системы в реальных промышленных условиях.

### 🔜 Ближайшие действия:
1. Утвердить концепцию архитектурным комитетом
2. Запустить параллельную разработку артефактов `[6]`, `[7]`, `[8]` (API, DB Model, Security)
3. Подготовить тестовый стенд с симуляторами Modbus/OPC UA
4. Сформировать команду и начать Спринт 1 (ядро + базовый адаптер + нормализация + API истории)
