# Интеграционный Слой (Integration Layer)
## Концептуальная спецификация

Интеграционный слой — это «переводчик» и «шлюз» между физическим миром устройств и логическим миром ядра системы. Его задача — сделать так, чтобы ядро *не знало и не заботилось* о том, как именно устроено устройство, а работало только с унифицированной моделью данных.

> **Ключевая метафора:** Интеграционный слой — это «универсальный адаптер розеток». Устройство может иметь любую «вилку» (протокол), но в «розетку» ядра всегда вставляется стандартный «штекер» (нормализованное событие).

---

## 1. Архитектура Адаптеров (Adapter Architecture)

### Принцип «Порт-Адаптер» (Hexagonal Architecture)
Каждое устройство или протокол подключается через изолированный модуль-адаптер, который реализует единый интерфейс:

```text
interface DeviceAdapter {
  connect(config) → bool           // Установить соединение
  read(tags[]) → [RawData]         // Опросить значения
  write(tag, value) → status       // (Опционально) записать команду
  subscribe(callback) → stream     // (Опционально) push-режим
  disconnect() → void              // Корректно закрыть соединение
  getDiagnostics() → HealthInfo    // Статус здоровья адаптера
}
```

### Жизненный цикл адаптера:
```
[Неактивен] 
    │ (загружена конфигурация)
    ▼
[Инициализация] → валидация параметров, подготовка ресурсов
    │
    ▼
[Подключение] → установка связи с устройством/сетью
    │
    ├─[Успех] → [Рабочий режим] ↔ циклы опроса / приём событий
    │               │
    │               ├─[Ошибка чтения] → локальная обработка → повтор
    │               └─[Критическая ошибка] → [Переподключение]
    │
    └─[Неудача] → [Режим ожидания] → экспоненциальная задержка → повтор
```

### Принципы:
1.  **Изоляция сбоев:** Падение или зависание одного адаптера не влияет на работу других.
2.  **Ленивая инициализация:** Адаптер запускается только когда есть конфигурация для него и потребность в данных.
3.  **Горячая замена:** Адаптер можно перезагрузить или переконфигурировать без остановки всего слоя интеграции.

## 2. Абстракция Протокола (Protocol Abstraction)

### Концепция «Виртуального канала связи»
Независимо от физического протокола (Modbus, OPC UA, MQTT, HTTP, Serial), адаптер работает с абстрактным каналом:

```text
VirtualChannel {
  open(params) → connection_id
  send(request) → future<response>
  receive() → stream<message>
  close(connection_id)
  
  // Мета-информация для диагностики
  stats: {
    latency: duration,
    errors: counter,
    last_seen: timestamp
  }
}
```

### Уровни абстракции протокола:

| Уровень | Что абстрагируется | Пример |
|---------|-------------------|--------|
| **Транспорт** | Физический способ доставки | TCP/UDP, Serial, WebSocket |
| **Кодирование** | Формат байт в сообщении | Binary, JSON, XML, Protobuf |
| **Семантика** | Логика запрос-ответ / pub-sub | Опрос регистра / подписка на топик |
| **Безопасность** | Механизмы аутентификации | Логин/пароль, сертификаты, токены |

### Принцип «Декларативного маппинга»:
Адаптер не содержит жёсткой логики под конкретное устройство. Вместо этого поведение описывается конфигурацией:

```yaml
# Конфигурация адаптера: Modbus -> Виртуальный тег
adapter:
  type: modbus_tcp
  connection:
    host: 192.168.1.100
    port: 502
    timeout: 2s
  
  mapping:
    - tag: "motor_temperature"
      source:
        register: 4001
        type: float32
        byte_order: big_endian
      transform:
        scale: 0.1        # raw_value * 0.1
        offset: -273.15   # конверсия в Цельсии
      validation:
        min: -50
        max: 200
```

→ Добавление нового устройства = новый YAML-файл, а не новый код.

## 3. Управление Соединениями (Connection Lifecycle Management)

### Принципы:
1.  **Пулл соединений (Connection Pooling):**
    *   Для протоколов с дорогим установлением соединения (например, база данных как источник) адаптер держит пулл готовых каналов.
    *   Запросы на чтение берут соединение из пула и возвращают его после выполнения.

2.  **Экспоненциальная отсрочка переподключения:**
    ```
    Попытка 1: ждать 1с
    Попытка 2: ждать 2с
    Попытка 3: ждать 4с
    ...
    Максимум: 60с
    ```
    → Предотвращает «шторм переподключений» при массовом сбое сети.

3.  **Health Check (проверка жизнеспособности):**
    *   Адаптер периодически отправляет лёгкий запрос (ping, чтение системного регистра) для проверки, что устройство «живо».
    *   Если устройство не отвечает — адаптер переводит теги в статус `quality=bad` и генерирует событие «потеря связи».

4.  **Грациозное завершение:**
    *   При остановке системы адаптер получает сигнал `shutdown`, завершает текущие запросы, сбрасывает буферы и закрывает соединения.

## 4. Обработка Ошибок и Восстановление (Error Handling Strategy)

### Классификация ошибок:

| Тип ошибки | Пример | Реакция адаптера |
|------------|--------|-----------------|
| **Транспортная** | Таймаут, разрыв сокета | Повторная попытка подключения по экспоненциальной схеме |
| **Протокольная** | Некорректный ответ, ошибка чек-суммы | Логирование, повтор запроса (max 3 раза), затем `quality=bad` |
| **Данных** | Значение вне диапазона, неверный тип | Валидация по правилам, установка `quality=uncertain`, запись «сырого» значения в лог |
| **Конфигурационная** | Неверный адрес регистра, несуществующий тег | Адаптер не запускается, ошибка валидации при загрузке конфигурации |
| **Критическая** | Исключение, нехватка памяти | Перезапуск адаптера, алерт в систему мониторинга |

### Принципы обработки:
1.  **Локализация сбоя:** Ошибка в одном теге не блокирует опрос других тегов того же устройства.
2.  **Прозрачность для ядра:** Ядро получает данные с флагом `quality`, а не исключение. Логика обработки качества — ответственность ядра.
3.  **Диагностический след:** Каждая ошибка логируется с контекстом: `{device_id, tag, timestamp, error_code, raw_payload}` для последующего анализа.

## 5. Безопасность на Границе (Edge Security)

### Принципы:
1.  **Минимальные привилегии:**
    *   Адаптер имеет доступ только к тем регистрам/топикам, которые явно указаны в конфигурации.
    *   Нет «чтения всего подряд» по умолчанию.

2.  **Изоляция учётных данных:**
    *   Пароли, токены, сертификаты хранятся в зашифрованном хранилище конфигурации, а не в открытом тексте.
    *   Адаптер получает доступ к секретам через защищённый API, а не читает файлы напрямую.

3.  **Валидация входящих данных:**
    *   Даже «доверенное» устройство может слать некорректные данные. Адаптер применяет базовую валидацию (тип, диапазон, частота) перед передачей в ядро.

4.  **Аудит действий:**
    *   Все операции записи команд на устройство (если адаптер поддерживает write) логируются: `{who, when, what_tag, old_value, new_value}`.

## 6. Расширяемость и Плагин-Модель (Extensibility)

### Концепция «Динамической загрузки драйверов»:
1.  **Контракт интерфейса:**
    *   Любой новый адаптер должен реализовать стандартный интерфейс `DeviceAdapter`.
    *   Ядро не знает внутренностей адаптера, только как его вызвать.

2.  **Реестр адаптеров:**
    ```text
    AdapterRegistry {
      register(type_name, adapter_factory)
      create(type_name, config) → DeviceAdapter
      list_available() → [type_names]
    }
    ```

3.  **Механизм обнаружения:**
    *   При старте система сканирует папку `plugins/` или использует механизм динамической линковки.
    *   Найденные модули регистрируются в реестре.

4.  **Версионирование адаптеров:**
    *   Адаптер указывает совместимую версию контракта: `adapter_api_version: "1.2"`.
    *   Ядро проверяет совместимость перед загрузкой.

### Сценарий добавления нового протокола:
```
1. Разработчик пишет класс-адаптер, реализующий интерфейс DeviceAdapter
2. Компонует его как динамическую библиотеку (или пакет)
3. Кладёт в папку plugins/ и добавляет описание в manifest.json
4. Перезагружает конфигурацию системы (или перезапускает сервис)
5. Новый тип адаптера доступен для использования в YAML-конфигурациях
```

## 7. Мониторинг и Диагностика Интеграций

### Метрики адаптера (экспортируются в ядро):
```text
adapter_metrics:
  connection:
    status: connected|disconnected|error
    uptime: duration
    reconnect_count: counter
  
  data_flow:
    messages_received: counter
    messages_sent: counter
    bytes_transferred: counter
    avg_latency_ms: gauge
  
  errors:
    transport_errors: counter
    protocol_errors: counter
    validation_errors: counter
  
  tags:
    total_count: gauge
    active_count: gauge
    bad_quality_count: gauge
```

### Принципы наблюдаемости:
1.  **Самодиагностика:** Адаптер знает своё состояние и сообщает его, а не ждёт опроса извне.
2.  **Структурированные логи:** Все события логируются в машиночитаемом формате (JSON) с обязательными полями: `timestamp, level, adapter_id, message, context`.
3.  **Трассировка запроса:** Каждому запросу от ядра к устройству присваивается `correlation_id`, который проходит через все слои. Это позволяет отследить полный путь: «запрос → адаптер → устройство → ответ → ядро».

## 8. Конфигурация Интеграционного Слоя

### Иерархия конфигурации:
```yaml
# Глобальные настройки слоя
integration:
  max_concurrent_adapters: 50
  default_timeout: 5s
  retry_policy:
    max_attempts: 3
    backoff_multiplier: 2
  
  # Список устройств
  devices:
    - id: "plc_conveyor_01"
      adapter_type: modbus_tcp
      enabled: true
      config: { ... } # специфично для адаптера
      tags: [ ... ]   # список тегов для этого устройства
      
    - id: "sensor_iot_42"
      adapter_type: mqtt
      config: { ... }
      tags: [ ... ]
```

### Принципы конфигурирования:
1.  **Наследование и переопределение:**
    *   Можно задать общие настройки для группы устройств и переопределить их для конкретного экземпляра.
2.  **Валидация при загрузке:**
    *   Конфигурация проверяется на наличие обязательных полей, корректность типов, ссылки на существующие адаптеры.
3.  **Динамическое обновление:**
    *   Изменение конфигурации устройства (добавление тега, смена интервала опроса) применяется без перезапуска адаптера.
4.  **Шаблонизация:**
    *   Для типовых устройств можно использовать шаблоны: `template: standard_pump_v2`, чтобы не дублировать конфигурацию.

## ✅ Критерии качества интеграционного слоя

| Критерий | Метрика / Признак |
|----------|------------------|
| **Прозрачность** | Добавление нового типа устройства требует только конфигурации или плагина, а не изменения ядра |
| **Надёжность** | Адаптер восстанавливается после сбоя связи без вмешательства человека |
| **Производительность** | Задержка адаптера (устройство→ядро) не превышает заданного порога (например, 50 мс для локальной сети) |
| **Безопасность** | Учётные данные не хранятся в открытом виде, доступ к устройствам минимально необходим |
| **Наблюдаемость** | По метрикам и логам можно точно определить, на каком этапе и почему произошла ошибка |
| **Масштабируемость** | Увеличение числа устройств в 10 раз требует только горизонтального добавления ресурсов, а не перепроектирования |

---

> 💡 **Философия слоя:** *«Устройство — это чёрный ящик с контрактом. Адаптер — это переводчик контракта. Ядро — потребитель переведённого смысла».*

Такая архитектура позволяет подключать любые устройства — от старого ПЛК с Modbus RTU до облачного IoT-шлюза с MQTT — без изменения логики ядра. Вся специфика «заземляется» в адаптерах, которые изолированы, заменяемы и конфигурируемы.

Если нужно — могу детализировать любой из подблоков: например, стратегию маппинга тегов, механизм горячего обновления конфигурации или паттерны тестирования адаптеров.
