# Техническое Задание (ТЗ) на разработку платформы IoT «IoT-Core»

## 1. Общие сведения
**Наименование проекта:** Платформа сбора, хранения и обработки телеметрии IoT.
**Цель:** Создание масштабируемой системы для приема данных с разнородных датчиков, их архивирования, предоставления доступа через API и автоматической реакции на события (триггеры) через вебхуки.
**Ключевое требование:** Микросервисная архитектура, модульность, возможность горизонтального масштабирования и расширения поддержки новых типов устройств без переписывания ядра системы.

## 2. Архитектурные принципы
Система строится на основе **событийно-ориентированной архитектуры (Event-Driven Architecture)**. Взаимодействие между сервисами осуществляется асинхронно через брокер сообщений.

1.  **Независимость сервисов:** Каждый модуль развертывается и масштабируется отдельно. Отказ одного модуля (кроме брокера сообщений) не должен приводить к полному останову системы.
2.  **Слабая связанность:** Сервисы не знают о внутренней реализации друг друга, общаются только через контракты (сообщения/события).
3.  **Расширяемость:** Поддержка новых протоколов и типов датчиков реализуется через систему плагинов (драйверов), не требующую изменения кода ядра.
4.  **Конфигурируемость:** Вся логика работы (правила, устройства, триггеры) хранится в БД и управляется через Админ-панель, без изменения кода и перезагрузки сервисов.

## 3. Состав системы (Микросервисы)

Система состоит из следующих независимых компонентов:

### 3.1. Service: IoT-Gateway (Шлюз сбора данных)
**Назначение:** Первичный прием данных от физических устройств.
**Функционал:**
*   Поддержка множественных протоколов связи (MQTT, HTTP/REST, CoAP, TCP Socket — расширяемо).
*   Валидация входящего пакета на базовом уровне (формат, подпись).
*   Нормализация данных: преобразование сырых данных в единый внутренний формат события.
*   Публикация нормализованных событий в Брокер Сообщений.
*   **Расширяемость:** Реализация через систему «Драйверов». Новый тип устройства добавляется загрузкой нового драйвера (модуля), описывающего логику парсинга и протокол.

### 3.2. Service: Data-Storage (Хранилище)
**Назначение:** Сохранение телеметрии и конфигурационных данных.
**Функционал:**
*   Подписка на события из Брокера Сообщений.
*   Запись телеметрии в специализированное хранилище (Time-Series DB) для быстрого чтения временных рядов.
*   Запись конфигураций (устройства, пользователи, правила) в реляционную БД.
*   Реализация политик хранения (TTL): автоматическое удаление старых данных или архивация в «холодное» хранилище.

### 3.3. Service: Rule-Engine (Движок правил и триггеров)
**Назначение:** Анализ данных в реальном времени и инициирование действий.
**Функционал:**
*   Подписка на поток телеметрии из Брокера Сообщений.
*   Оценка входящих данных против активных правил (триггеров), загруженных из кэша/БД.
*   Логика триггеров: пороговые значения, отсутствие данных (heartbeat), сложные условия (A > 10 AND B < 5).
*   При срабатывании триггера: формирование события и отправка запроса на внешний Вебхук.
*   Механизм повторных попыток (Retry Policy) при неудачной отправке вебхука.
*   Лимитирование частоты срабатывания (Throttling), чтобы не «заспамить» внешнюю систему.

### 3.4. Service: API-Gateway (Внешний API)
**Назначение:** Предоставление доступа к данным для внешних клиентов и фронтенда.
**Функционал:**
*   Единая точка входа для всех внешних запросов.
*   Авторизация и аутентификация (API Keys, JWT).
*   Маршрутизация запросов к внутренним сервисам (Storage, Admin).
*   Предоставление эндпоинтов для:
    *   Получения истории показаний датчиков.
    *   Получения статуса устройств.
    *   Управления конфигурацией (если не через Админ-панель).
*   Rate Limiting (ограничение частоты запросов).

### 3.5. Service: Admin-Panel (Веб-интерфейс и Backend)
**Назначение:** Управление конфигурацией системы.
**Функционал:**
*   Управление пользователями и правами доступа (RBAC).
*   **Реестр устройств:** Добавление, редактирование, удаление датчиков. Привязка драйвера к устройству.
*   **Настройка триггеров:** Визуальный или конструкторский интерфейс для создания правил (Если `Датчик Х` > `Значение Y`, то `Вызвать Вебхук Z`).
*   **Управление вебхуками:** Хранение URL, заголовков, методов отправки.
*   **Мониторинг:** Просмотр логов срабатывания триггеров, статусов доставки вебхуков, состояния устройств (онлайн/офлайн).

### 3.6. Infrastructure: Message Broker (Брокер сообщений)
**Назначение:** Связующее звено между сервисами.
**Требования:** Гарантированная доставка сообщений, поддержка очередей, возможность масштабирования кластера. (Техническая реализация выбирается на этапе проектирования, например, Kafka, RabbitMQ или NATS).

## 4. Логика работы и потоки данных

### 4.1. Поток приема данных (Ingestion Flow)
1.  Датчик отправляет данные по своему протоколу в **IoT-Gateway**.
2.  **IoT-Gateway** определяет тип устройства, применяет соответствующий Драйвер, преобразует данные в стандартный JSON-событие.
3.  Событие публикуется в топик Брокера Сообщений `telemetry.raw`.
4.  **Data-Storage** забирает событие и сохраняет в БД.
5.  **Rule-Engine** забирает событие и проверяет условия триггеров.

### 4.2. Поток срабатывания триггера (Trigger Flow)
1.  **Rule-Engine** обнаруживает соответствие условия триггера.
2.  Создается событие `trigger.fired`.
3.  **Rule-Engine** (или отдельный воркер) выполняет HTTP-запрос на настроенный Вебхук.
4.  Результат выполнения (успех/ошибка) логируется в **Admin-Panel** для просмотра администратором.

### 4.3. Поток запроса данных (Query Flow)
1.  Внешняя система отправляет запрос в **API-Gateway** (например, «дай данные за час»).
2.  **API-Gateway** проверяет права доступа.
3.  Запрос перенаправляется в **Data-Storage**.
4.  Данные возвращаются клиенту.

## 5. Требования к расширяемости (Экосистема датчиков)

Для поддержки новых типов датчиков без изменения ядра системы реализуется **Интерфейс Драйвера (Driver Interface)**.

*   **Контракт:** Каждый драйвер должен реализовывать стандартный набор методов (например: `connect`, `parse`, `validate`, `getMetrics`).
*   **Реестр драйверов:** В Админ-панели при создании устройства пользователь выбирает тип драйвера из списка доступных.
*   **Механизм загрузки:** Драйверы могут быть встроены в код сервиса Gateway или загружаться как динамические модули/скрипты (в зависимости от выбранного стека), что позволяет обновлять поддержку устройств независимо от релиза платформы.
*   **Маппинг данных:** Драйвер должен уметь мапить специфичные поля устройства на унифицированные метрики системы (например, поле `temp_val` с устройства -> метрика `temperature` в системе).

## 6. Требования к Админ-панели

Админ-панель должна быть SPA (Single Page Application) и предоставлять следующий функционал:
1.  **Дашборд:** Общая статистика (кол-во устройств, сообщений в секунду, активных триггеров).
2.  **Устройства:** Список, фильтрация, поиск, карта (опционально), детальная карточка устройства (настройка интервала опроса, привязка драйвера).
3.  **Конструктор триггеров:**
    *   Выбор устройства/метрики.
    *   Выбор оператора (> , < , =, range, absence).
    *   Задание значения.
    *   Выбор действия (Вебхук).
    *   Настройка «окна» срабатывания (чтобы не срабатывало на единичный шум).
4.  **Логи:** Журнал событий (кто зашел, какое правило сработало, какая ошибка при отправке вебхука).
5.  **Настройки системы:** Управление доступом, общие параметры таймаутов и т.д.

## 7. Нефункциональные требования

1.  **Производительность:**
    *   Система должна поддерживать прием данных от N тысяч устройств одновременно (число уточняется на этапе проектирования инфраструктуры).
    *   Задержка от получения данных до срабатывания триггера не должна превышать X секунд (Near Real-time).
2.  **Надежность:**
    *   При падении сервиса Storage данные не должны теряться (должны сохраняться в Брокере сообщений до восстановления).
    *   Механизм Dead Letter Queue (DLQ) для сообщений, которые не удалось обработать.
3.  **Безопасность:**
    *   Все внешние коммуникации (API, Вебхуки, Датчики) только по защищенным протоколам (HTTPS, TLS).
    *   Хранение секретов (ключи API, пароли БД) в защищенном хранилище (Secrets Manager), не в коде.
    *   Изоляция сетей: сервисы БД не должны иметь прямого доступа из интернета.
4.  **Масштабируемость:**
    *   Возможность горизонтального масштабирования сервисов Gateway и Rule-Engine путем добавления новых инстансов (контейнеров).
    *   Stateless архитектура сервисов (состояние хранится в БД/Брокере, а не в памяти сервиса).

## 8. Этапы разработки (Roadmap)

1.  **Прототипирование:** Утверждение архитектуры, выбор стека технологий, проектирование схемы БД и форматов сообщений.
2.  **Ядро (MVP):** Реализация Брокера, Storage, базового Gateway (HTTP/MQTT) и простого API.
3.  **Логика:** Разработка Rule-Engine и механизма Вебхуков.
4.  **Администрирование:** Разработка Админ-панели для настройки.
5.  **Расширяемость:** Реализация системы Драйверов для подключения кастомных устройств.
6.  **Тестирование и Нагрузка:** Проверка на отказоустойчивость и пиковые нагрузки.

## 9. Ограничения и допущения
*   Система не управляет напрямую «железом» датчиков (не прошивает их), только принимает данные.
*   Поддержка видео-потоков или тяжелых бинарных файлов не входит в текущую версию (только телеметрия/метаданные).
*   Внешние вебхуки считаются ненадежными, система должна корректно обрабатывать их недоступность.
