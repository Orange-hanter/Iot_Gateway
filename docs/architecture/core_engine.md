# Принципы работы Ядра Системы (Core Engine)
## Концептуальная спецификация

Ядро системы — это «мозг», который находится между физическим миром (устройства) и бизнес-логикой (MES, аналитика, люди). Его главная задача — превратить поток сырых сигналов в структурированную, надёжную и осмысленную информацию.

Ниже максимально подробно расписаны принципы работы каждого внутреннего механизма ядра, без привязки к конкретному стеку технологий.

---

## 1. Единая Модель Данных (Universal Data Model)
**Проблема:** Устройства говорят на разных языках (регистры, JSON, биты, строки).
**Решение:** Ядро приводит всё к единому стандарту сразу на входе.

### Концепция «Нормализованного Тега»
Любая входящая точка данных внутри ядра существует только в формате:
```text
{
  "identity": {
    "device_id": "unique_identifier",
    "tag_name": "logical_name",      // Человекочитаемое имя
    "data_type": "numeric|boolean|string"
  },
  "payload": {
    "value": <any>,                  // Нормализованное значение
    "timestamp": "ISO8601_utc",      // Единое время (всегда UTC)
    "quality": "good|uncertain|bad"  // Код качества данных
  }
}
```

### Принципы обработки:
1.  **Типизация:** Если устройство прислало строку "100", ядро преобразует её в число `100` согласно конфигурации тега. Если не может — ставит качество `bad`.
2.  **Временная синхронизация:**
    *   Если устройство имеет часы: используется время устройства.
    *   Если часов нет: используется время прибытия в ядро + известная задержка сети.
    *   *Коррекция дрейфа:* Ядро отслеживает рассинхронизацию часов устройства и применяет поправку.
3.  **Коды качества (Quality Codes):** Критически важно для IIoT. Если связь с датчиком потеряна, ядро не просто перестает писать данные, а генерирует событие `quality=bad`. Это позволяет триггерам отличать «нормальное значение 0» от «нет данных».

---

## 2. Асинхронный Конвейер (Asynchronous Pipeline)
**Проблема:** Сбор данных не должен ждать, пока сохранится история или сработает тревога.
**Решение:** Разделение потоков обработки (Separation of Concerns).

### Этапы конвейера:
1.  **Ingestion (Приём):** Только приём и валидация. Максимально быстро. Никакой сложной логики.
2.  **Dispatch (Маршрутизация):** Данные копируются в несколько независимых потоков:
    *   Поток А → В буфер хранения.
    *   Поток Б → В движок триггеров.
    *   Поток В → В кэш «последнего значения» (для быстрого API).
3.  **Изоляция ресурсов:** Если движок триггеров «завис» на сложном вычислении, сбор данных продолжается. Буфер накапливает данные, чтобы отдать их на обработку позже.

---

## 3. Движок Триггеров и Состояний (Stateful Rule Engine)
**Проблема:** Триггеры — это не просто `if value > 10`. Это часто `if value > 10 for 5 minutes`.
**Решение:** Управление состоянием правил.

### Принципы работы:
1.  **Оконная агрегация (Windowing):**
    *   Ядро хранит «скользящее окно» данных для каждого правила.
    *   Пример: Для правила «Средняя скорость за 1 минуту» ядро держит в памяти буфер последних 60 секунд данных по этому тегу.
2.  **Машина состояний правила (Rule State Machine):**
    *   Каждое правило имеет статус: `Inactive` → `Pending` → `Active` → `Acknowledged`.
    *   *Сценарий:* Температура превысила норму.
        1.  Условие выполнено → Статус `Pending` (запуск таймера).
        2.  Условие держится 30 сек → Статус `Active` (генерация события/алерта).
        3.  Условие пропало → Статус `Recovered` (событие восстановления).
        4.  Оператор нажал «Принято» → Статус `Acknowledged`.
3.  **Приоритизация:** Критические алерты (пожар) обрабатываются вне очереди, игнорируя нагрузку. Информационные (план выполнен) могут быть отложены.
4.  **Дедупликация событий:** Ядро не спамит. Если условие «Авария» истинно, событие генерируется *один раз* при переходе в состояние, а не при каждом цикле опроса.

---

## 4. Стратегия Хранения (Storage Strategy)
**Проблема:** Данные пишутся очень часто (секунды), а читаются редко (отчеты).
**Решение:** Оптимизация под запись (Write-Optimized) и иерархия данных.

### Принципы:
1.  **Разделение Hot/Cold данных:**
    *   **Hot (Оперативные):** Последние N часов/суток. Хранятся в быстром слое (например, в памяти или SSD) для мгновенного доступа в дашборды.
    *   **Cold (Архив):** Старые данные. Сжимаются и перекладываются в долгосрочное хранилище.
2.  **Политики Ретенции (Retention Policies):**
    *   Конфигурируемое правило: «Хранить сырые данные 7 дней, затем удалять».
    *   Конфигурируемое правило: «Каждые 15 минут вычислять среднее/мин/макс и хранить агрегаты 5 лет».
3.  **Идемпотентность записи:**
    *   Если данные пришли дважды (например, после восстановления связи), ядро определяет дубликат по уникальному ключу `{device_id, tag, timestamp}` и не создаёт вторую запись.
4.  **Пакетная запись (Batching):**
    *   Ядро не пишет каждую точку сразу на диск. Оно накапливает пакет (например, 1000 точек или 1 секунда) и пишет одним транзакционным блоком. Это снижает нагрузку на подсистему хранения в десятки раз.

---

## 5. Надёжность и Отказоустойчивость (Resilience)
**Проблема:** Сбои питания, сети, перезагрузки ядра.
**Решение:** Гарантированная доставка и восстановление состояния.

### Принципы:
1.  **Write-Ahead Log (Журнал упреждающей записи):**
    *   Прежде чем обработать данные, ядро записывает их в надёжный журнал на диск. Если система упадет в процессе обработки, при перезагрузке она «проиграет» журнал и не потеряет данные.
2.  **Буферизация при разрыве связи (Store & Forward):**
    *   Если ядро не может отправить данные в верхний уровень (облако/центральный сервер MES), оно хранит их локально в очерёди.
    *   При восстановлении связи отправляет пакетом, сохраняя оригинальные временные метки.
3.  **Graceful Degradation (Деградация функционала):**
    *   Если хранилище истории недоступно, ядро не падает. Оно продолжает собирать данные в оперативную память и обслуживать триггеры, временно отключая запись в архив.
4.  **Watchdog (Сторожевой таймер):**
    *   Внутренний мониторинг здоровья процессов. Если модуль сбора «завис», он автоматически перезапускается без остановки всего ядра.

---

## 6. Управление Конфигурацией (Configuration Management)
**Проблема:** Изменение логики не должно требовать перезагрузки или пересборки.
**Решение:** Динамическая подгрузка конфигурации.

### Принципы:
1.  **Hot Reload (Горячая перезагрузка):**
    *   Инженер меняет файл конфигурации (добавляет новый датчик или меняет порог тревоги).
    *   Ядро detect-ит изменение, валидирует новую конфигурацию и применяет её «на лету».
    *   Существующие сборщики данных не прерываются, новые запускаются параллельно.
2.  **Версионирование конфигурации:**
    *   Ядро хранит историю изменений конфигурации. Если новая настройка привела к ошибкам, возможен откат к предыдущей версии.
3.  **Валидация схем:**
    *   Перед применением конфигурация проверяется на логические ошибки (например, «триггер ссылается на несуществующий тег»).

---

## 7. Интерфейс Взаимодействия (API & Integration)
**Проблема:** Внешние системы (MES, ERP, Dashboards) хотят данные по-разному.
**Решение:** Абстрактный слой доступа к данным.

### Принципы:
1.  **Разделение путей Чтения и Записи (CQRS concept):**
    *   Путь записи (от устройств) оптимизирован для скорости.
    *   Путь чтения (запросы истории) оптимизирован для выборки диапазонов и агрегации.
2.  **Подписка на события (Pub/Sub):**
    *   Внешние системы могут не опрашивать ядро («Есть новые данные?»), а подписаться на поток («Сообщай мне, когда придет новое значение тега Х»).
    *   Ядро само пушит данные подписчикам.
3.  **Контроль доступа (Scope & Quotas):**
    *   Разные клиенты имеют разные права. Оператор видит всё, сторонняя система — только конкретные теги.
    *   Лимитирование запросов, чтобы «тяжёлый» отчет не положил систему сбора.

## ✅ Критерии успешности работы Ядра

1.  **Determinism (Детерминизм):** При одинаковых входных данных и конфигурации ядро всегда выдаст одинаковые события и записи.
2.  **Latency (Задержка):** Время от появления сигнала на устройстве до записи в БД и срабатывания триггера должно быть предсказуемым (например, < 100 мс внутри локальной сети).
3.  **No Data Loss (Отсутствие потерь):** При корректной работе оборудования ни один бит данных не теряется внутри ядра.
4.  **Scalability (Масштабируемость):** Добавление 1000 новых тегов не должно требовать изменения архитектуры, только расширения ресурсов (CPU/RAM).

Этот набор принципов позволяет создать ядро, которое будет работать одинаково стабильно и на маленьком конвейере, и на крупном заводе, просто за счёт изменения конфигурации и выделения ресурсов.
