# Архитектура модуля сбора данных для MES/IIoT
## Концептуальный проект (уровень абстракции: модули и потоки)

---

## 🎯 Ключевые принципы проектирования

| Принцип | Зачем нужен |
|---------|-------------|
| **Абстракция от устройства** | Не важно, какой протокол или тип устройства — система работает с унифицированной моделью данных |
| **Модульность** | Каждый компонент можно заменить или масштабировать независимо |
| **Конфигурируемость** | Интеграция на новом объекте — это настройка, а не переписывание кода |
| **Событийная модель** | Триггеры и оповещения обрабатываются асинхронно, не блокируя сбор данных |
| **Локальность + облачность** | Ядро работает на edge-устройстве, но готова к синхронизации с центром |

## 🔁 Поток данных (упрощённо)

1. **Устройство → Адаптер**  
   - Опрос или push-режим  
   - Преобразование в внутреннюю структуру: `{device_id, tag, value, timestamp, quality}`

2. **Адаптер → Ядро**  
   - Данные попадают в надёжную очередь (гарантия доставки при сбоях)  
   - Параллельно: запись в хранилище + проверка триггеров

3. **Обработчик триггеров**  
   - Правила описываются декларативно:  
     ```
     IF temperature > 80 AND duration > 30s 
     THEN trigger "overheat" WITH severity=high
     ```
   - События триггеров → подсистема оповещений

4. **Хранилище**  
   - Оптимизировано под временные ряды  
   - Поддержка политик хранения: "сырые данные 7 дней → агрегаты 1 год"

5. **Внешние запросы**  
   - API для получения истории, статусов, метрик  
   - Аутентификация и лимитирование запросов

## 🧩 Точки расширения (как масштабировать)

| Компонент | Как расширять |
|-----------|--------------|
| **Адаптеры устройств** | Добавлять новые драйверы без изменения ядра (плагин-модель) |
| **Правила триггеров** | Подключать внешние движки правил или скрипты (Lua/JS) |
| **Хранилище** | Абстракция слоя данных: замена БД не требует переписывания логики |
| **Каналы оповещений** | Новые интеграции (Telegram, SCADA, ERP) через конфигурацию |
| **Вычислительные узлы** | Горизонтальное масштабирование: несколько edge-узлов → один центр |

## ⚙️ Конфигурация вместо кода

Для быстрой интеграции на новом предприятии вся логика настраивается через декларативные файлы:

```yaml
# Пример: точка сбора
device:
  id: conveyor_01
  adapter: modbus_tcp
  poll_interval: 2s
  tags:
    - name: speed
      address: 4001
      type: float
    - name: status
      address: 4002
      type: int

# Пример: триггер
trigger:
  name: low_speed_alert
  condition: "speed < 10 AND status == 1"
  window: 10s
  actions:
    - notify: "operator"
    - log: "maintenance_queue"
```

→ Инженер на месте правит YAML/JSON, а не код.

## 🔐 Надёжность и эксплуатация

- **Буферизация**: при потере связи данные сохраняются локально и отправляются позже
- **Идемпотентность**: повторная отправка данных не создаёт дубликатов
- **Мониторинг**: метрики здоровья модулей (heartbeat, задержки, ошибки)
- **Обновления**: модули обновляются независимо, без остановки всего сбора

## 🚀 Что дальше: следующие уровни детализации

Когда концепция утверждена, можно углубляться по направлениям:

1. **Модель данных**: единая схема тега, единицы измерения, качества данных
2. **Протоколы адаптеров**: приоритетный список (Modbus, OPC UA, MQTT, HTTP)
3. **Движок правил**: выбор между простым DSL и внешним движком (Drools, CEL)
4. **Стратегия хранения**: retention-политики, сжатие, агрегация
5. **Безопасность**: аутентификация устройств, шифрование каналов, аудит

---

> 💡 **Ключевая мысль**: система проектируется не под устройства, а под *интерфейсы взаимодействия*. Устройство — это источник данных с контрактом. Всё остальное — конфигурация.

Если нужно — могу детализировать любой из блоков (например, модель триггеров или стратегию синхронизации edge↔cloud) или подготовить чеклист для пилотной интеграции на объекте.
